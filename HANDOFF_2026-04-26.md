### 1. QA review fixes applied to existing modules

**`pool_curator.py`**
- `CuratedPool.to_dict()` now persists `win_rate`, `confirm_action_per_game`, and `suspected_inflated` per score record. Previously `asdict()` silently dropped these because they're `@property` fields, so the persisted pool was missing the inflation signal the curator was supposed to surface.

**`moxfield_import.py`**
- `_uniquify()` now raises instead of silently overwriting after 99 collisions (the bug it exists to prevent).
- Removed unused `sys` and `Iterable` imports.

**`log_parser.py`**
- **Real bug fixed**: `_normalize()` was stripping in the wrong order — the `[B<n>]$` regex couldn't match because `.dck` was still trailing. Result: `[USER] Foo [B3].dck` → `Foo [B3]` instead of `Foo`. This silently broke `_filename_for_match` for any decorated input. Now `.dck` is stripped first, then the bracket. ### 2. New module: `game_analyzer.py`

Sits on top of `log_parser` and extracts richer per-game telemetry that wasn't being captured:

- End turn (`Game Outcome: Turn N`)
- Authoritative winner (`Game Result: ... has won!` — the trailing clause is reliable, unlike the 4× `Game Outcome:` lines that all say "won" in 4-player commander)

- Per-deck life curves: starting/ending/min/max life, damage taken, life gained
- Elimination turn (when life first hit 0)
- Per-game `confirmAction` count

Aggregates across games into `MatchAnalysis.per_deck_summary()`:
- `wins`, `eliminations`, `fastest_elimination_turn`, `avg_ending_life`, `total_damage_taken`, `total_life_gained`

### 3a. New module: `compare_versions.py`

**Old-vs-new deck comparison** — built per your suggestion ("run a 1v1 with the old deck playing the new deck"). Two modes:

- **`pod` (default, recommended)**: 4-player same-table comparison. Both versions sit in one pod with 2 fillers, facing the same opposition simultaneously. Direct head-to-head signal that preserves multiplayer dynamics (political AI, archenemy targeting). Cheaper than a 1v1 series — 1 sim run per pod.
- **`1v1`**: Forge constructed format, 2 decks per sim. Pure efficiency comparison. Singleton/color-identity rules not enforced — fine for same-skeleton comparisons.

Filler-pair rotation (`--filler-pairs 2`, default): runs the comparison twice with two different filler pairs from the bracket pool, averaging out single-pair bias. Aggregates wins across both runs.

Outputs a `ComparisonReport` with:
- Head-to-head record (`OLD 4 - 6 NEW`), `winner` ("old"/"new"/"tie"), `margin`
- Per-version `VersionStats` (wins, eliminations, avg ending life, avg damage taken, avg turns when winning vs losing, fastest elimination)
- Card-level diff (`added` / `removed` from the [Main] section, parsed straight from the .dck files)
- Per-pod `MatchAnalysis` for post-mortem

CLI:
```
python -m commander_builder.compare_versions \
    --old "[USER] My Deck v1 [B3].dck" \
    --new "[USER] My Deck v2 [B3].dck" \
    --bracket 3 --games 10 --filler-pairs 2
```

Realistic cost: ~30 min for 2 pods × 10 games at 4-player B3 pace (80s/game). Below 10 games, swap-level signal washes out in pod variance.

Persists `_compare/<old>__vs__<new>_B<n>_<timestamp>.json`.

### 3b. New module: `run_match.py`

The user-deck matchup orchestrator (Layer 2 of the testing strategy in PROJECT.md). Pits a `[USER]`-tagged deck against the curated pool (or falls back to first N on-disk opponents if no pool exists yet) and produces a `MatchupReport` with weakness signals:

- W/L/D record + win rate
- Avg ending life when winning vs damage taken per game
- Avg turns when winning vs losing
- Fastest elimination (consistency hole indicator)
- Per-opponent record (which decks beat the user, which fell)
- `unsupported_cards` list (concrete swap candidates)

CLI:
```
python -m commander_builder.run_match \
    --user "[USER] Hakbal of the Surging Soul [B3].dck" \
    --bracket 3 --games 3 --pods 2
```

Persists to `_matches/<deck>_B<n>_<timestamp>.json`.

## Known gaps still to address (from QA review)

These are documented but not yet fixed — flagging so a future session can pick up:

1. **`_filename_for_match` collision-suffix gap** — when a deck name is uniquified (`Foo (2) [B3].dck`) or when the internal `Name=` has Unicode the filename strips, the exact-stem match returns None. Pinned by `test_filename_for_match_documents_collision_suffix_gap`. Fix: switch to `_normalize`-based fuzzy lookup or update `to_dck` to write a sanitized internal name.

2. **`confirm_action_cards` not deck-attributed** — Forge logs the card but not the active player. Stopgap divides by pod size; proper fix needs `log_parser` to track "active player" context from nearby `Phase: Ai(N)-...` lines.

3. **`_split_into_slices` one-shot swap can leave persistent violations** — if swap of ranks 3↔4 fixes slice A but breaks slice B, no further check fires. Tolerable for now (rare with 5+ archetypes); revisit if violations persist.
